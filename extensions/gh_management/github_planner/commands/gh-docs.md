# /th:gh-docs

<!-- RULE: This command lives in github_planner for simplicity — no dedicated extension.
     It is intentionally named without "plan" since it runs post-implementation. -->

<!-- LOAD ANNOUNCEMENT: At the very start of this command, output exactly:
     🟢 Loaded: gh-docs — `extensions/gh_management/github_planner/commands/gh-docs.md`
     Do this before any tool calls. -->

Post-implementation doc writer — creates or updates user and developer docs, then opens a PR or pushes directly.

---

## When to use

- User runs `/th:gh-docs` explicitly
- `gh-implementation` closes the last open issue → prompt: "All issues are closed. Want me to write or update docs? (`/th:gh-docs`)"

---

## Flow

### Step 0 — Load docs guide (always first)

Check for `hub_agents/docs_guide.md`:

- **Exists** → read it silently. Apply all preferences throughout generation. Do not mention it unless the user asks.
- **Does not exist** → create it with the default scaffold below, then say exactly:
  > "I've created `hub_agents/docs_guide.md` — edit it anytime to control how your docs are written."

**Default scaffold:**
```markdown
# Docs Guide

Preferences applied by /th:gh-docs when writing or updating README.md and CONTRIBUTING.md.
Edit this file directly or tell Claude your preferences during a /th:gh-docs session.

## General
- Language: English
- Tone: concise and direct

## README.md
- (add preferences here)

## CONTRIBUTING.md
- (add preferences here)
```

### Step 1 — Ask which docs to generate

Ask: "Which docs should I write or update? **(both / user / dev)**"

- **both** → generate README.md + CONTRIBUTING.md
- **user** → generate README.md only
- **dev** → generate CONTRIBUTING.md only

### Step 2 — Check for existing files

Before writing, check if the target files already exist:

```
README.md       → if exists: read it first, patch (do not overwrite structure the user has customised)
CONTRIBUTING.md → if exists: read it first, patch (extend sections, do not duplicate)
```

If a file does not exist: create it fresh using the full structure from the skill.

### Step 3 — Load skill(s) and generate

- **User docs** → call `load_skill("create_user_readme_docs")` → follow skill exactly
- **Dev docs** → call `load_skill("create_dev_readme_docs")` → follow skill exactly

Apply all preferences from `docs_guide.md` while generating. Do not write to disk yet — hold content in context.

### Step 4 — Show a brief preview

Show a short preview (headings only) of what will be written or changed. Ask: "Look good? (yes / edit)"

On "edit": accept corrections, regenerate, ask again.

### Step 5 — Write files to disk

Write the final content to `README.md` and/or `CONTRIBUTING.md` in the project root.

### Step 6 — PR or push?

Ask: "Docs are ready. Want me to open a PR for review (I'll share the link) or push directly to main?"

**PR path:**
```bash
git checkout -b docs/update-$(date +%Y%m%d)
git add README.md CONTRIBUTING.md
git commit -m "docs: update user and developer documentation"
git push -u origin HEAD
gh pr create --title "docs: update user and developer documentation" \
  --body "Generated by /th:gh-docs — updates README.md and/or CONTRIBUTING.md."
```
→ Return the PR URL. Say: "PR open — review it here: {url}"

**Push path:**
```bash
git add README.md CONTRIBUTING.md
git commit -m "docs: update user and developer documentation"
git push
```
→ Confirm: "Pushed to main."

## Conversational preference updates

During any `/th:gh-docs` session, watch for signals like:
- "for the docs, can we..."
- "make it more / less..."
- "don't include X"
- "always use Y"
- "keep it under N lines"

When detected:
1. Apply the preference immediately in the current generation
2. Append it to the relevant section in `hub_agents/docs_guide.md` silently
3. Confirm in one line: `"Got it — saved to docs_guide.md."`
4. **Never ask** "should I save this?" — just save
5. **Never re-prompt** a preference already in `docs_guide.md` — apply it silently on all future runs

---

## Rules

- Never overwrite a section the user has customised without showing a diff first
- Only stage `README.md` and `CONTRIBUTING.md` — do not `git add .`
- If `gh` CLI is not authenticated, say: "Run `gh auth login` first, then re-run `/th:gh-docs`"
- If no changes were made to an existing file, skip staging it
- Always run Step 0 before anything else — `docs_guide.md` is the source of truth for preferences
- Once a preference is saved to `docs_guide.md`, treat it as permanent until the user explicitly removes it
